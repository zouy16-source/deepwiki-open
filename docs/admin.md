## 1. 项目概述

### 1.1 项目简介

使用 AI 辅助业务需求落地，减少沟通成本，提升研发效率。平台将「业务 → 产品 → 开发 → 测试 → 验收 → 上线」全流程打通，以 AI Agent 为核心驱动力，把目前各角色分散使用的 AI 能力串联成一条完整的智能研发流水线，替代传统 TAPD 驱动的人工流转模式。

### 1.2 项目背景

当前研发团队中，产品、研发、测试均已各自使用 AI 辅助工作，但：

- AI 能力是**孤岛式**的，各角色各用各的，产出无法互相流转与复用；
- 需求流转仍依赖 TAPD + 人工沟通，AI 的分析结果无法沉淀为流程资产；
- 需求可行性判断依赖开发人员人工阅读存量业务代码，成本高、周期长。

### 1.3 项目目标

| 目标 | 度量指标（上线 6 个月后） |
| --- | --- |
| 降低需求沟通成本 | 需求澄清平均耗时下降 ≥ 50% |
| 缩短需求交付周期 | 需求从提出到上线的平均周期缩短 ≥ 30% |
| 提升可行性评估效率 | AI 可行性分析报告产出时间 ≤ 30 分钟（现状：1~3 天人工评估） |
| 流程全线上化 | 90% 以上需求全程在平台内流转，不再依赖 Excel/TAPD 双轨 |
| AI 产出可用率 | AI 生成的需求文档/测试用例经人工修订后采纳率 ≥ 70% |

### 1.4 项目价值

1. **对业务方**：需求提出即可获得可行性、复杂度、排期的快速反馈，不再"石沉大海"。
2. **对产品经理**：AI 代替人工做代码可行性摸底与需求文档初稿，聚焦在需求本身的价值判断。
3. **对研发**：需求带着完整上下文（关联代码、系统范围、设计初稿）到达，减少反复澄清；AI 辅助落地代码。
4. **对测试**：测试用例与自动化脚本由 AI 依据需求与代码自动生成，聚焦异常场景与验收把关。
5. **对管理层**：需求全生命周期数据沉淀，研发效能可度量、可分析。

## 2. 现状分析

### 2.1 现有需求研发流程

1. 业务方通过 Excel 表格向产品经理提出业务需求。
2. 业务方与产品经理沟通业务需求细节。
3. 产品经理与开发人员沟通已有的业务逻辑是否与新需求有冲突，是否可落地。
4. 产品经理答复业务方需求是否可落地、落地时间。
5. 产品经理编写业务需求与计划，在 TAPD 新增需求计划。
6. 业务需求开始评审，安排需求与计划的相关开发/测试，将 TAPD 分发给各个项目研发人员。
7. 研发人员根据 TAPD 需求开始开发。
8. 开发完成提交测试人员，测试人员开始测试。
9. 测试通过，提交业务方验收。
10. 验收通过，发布生产，需求流程结束。

### 2.2 现状痛点

| # | 痛点 | 根因 | 影响 |
| --- | --- | --- | --- |
| P1 | 沟通成本大，产品经理需反复确认需求可行性，需开发人工阅读存量业务代码来判断需求是否合理 | 业务逻辑知识只存在于代码和开发人员脑中，没有可查询的知识层 | 可行性评估周期长（1~3 天），占用核心开发人力 |
| P2 | 研发流水线未打通：产品/研发/测试各自使用 AI 辅助自己的工作，但流程仍靠传统 TAPD 流转 | 缺少统一平台承载 AI 产物与流程状态 | AI 分析结果无法复用；上下游信息在流转中衰减 |
| P3 | 需求以 Excel 提出，格式不一、信息缺失 | 无结构化需求入口 | 需求澄清轮次多，遗漏与返工频发 |
| P4 | 需求进展、效能数据分散在 TAPD/IM/口头 | 无统一数据底座 | 管理层无法量化研发效能与瓶颈 |

### 2.3 目标流程（To-Be）

```
业务方提交结构化需求
      │
      ▼
AI 可行性分析（业务可行性 / 代码可行性 / 复杂度 / 系统范围）
      │
      ▼
线上评审会（基于 AI 分析报告）──不通过──▶ 打回补充/关闭
      │ 通过
      ▼
AI 生成系统需求与设计 → 产品经理修订确认
      │
      ▼
AI 生成需求计划（主/子需求）→ 录入需求计划表 → 关联开发/测试人员
      │
      ▼
AI Agent 辅助编码落地（云开发工作台）→ 开发人员审查/补充 → 代码合入
      │
      ▼
AI 生成测试用例 + 自动化测试执行 → 测试人员把关
      │ 通过
      ▼
自动部署 UAT/PRE 环境 → 业务方线上验收
      │ 通过
      ▼
发布生产 → 需求关闭 → 数据沉淀入效能报表
```

## 3. 用户角色与权限

| 角色 | 主要职责 | 核心操作 |
| --- | --- | --- |
| 业务方（需求提出人） | 提出需求、参与评审、UAT 验收 | 提交需求、查看 AI 分析报告、验收确认 |
| 产品经理 | 需求管理、主持评审、确认 AI 产出 | 修订系统需求/设计、创建需求计划、分派人员 |
| 开发人员 | 代码实现、审查 AI 生成代码 | 云开发工作台、代码审查、合并请求 |
| 测试人员 | 测试把关、维护用例资产 | 审核 AI 用例、执行/查看自动化测试、缺陷管理 |
| 研发负责人 | 排期决策、资源调度 | 排期确认、效能报表 |
| 平台管理员 | 系统配置、权限管理 | 用户/角色/项目管理、Agent 与模型配置、代码库接入 |

权限模型采用 RBAC + 项目维度隔离：用户 ∈ 角色 ∈ 项目空间；跨项目数据默认不可见，代码知识库按项目授权访问。

## 4. 功能需求

需求编号规则：`FR-<模块>-<序号>`。优先级：P0（首版必须）/ P1（重要）/ P2（增强）。

### 4.1 模块一：需求管理中心

> **定位（已决策）**：本模块**完整替代 TAPD**。需求流转不再依赖外部工具，而由平台内置的状态机承载，且每个流转节点与 AI 产物（分析报告、生成文档、代码变更、测试结果）原生绑定——这是"更适合 AI 辅助流转"的关键差异：流转的不只是状态，还有完整上下文。

| 编号 | 需求 | 优先级 | 说明 |
| --- | --- | --- | --- |
| FR-REQ-01 | 结构化需求提交 | P0 | 业务方通过表单/模板提交需求（背景、目标、期望上线时间、涉及业务域），支持附件（Excel/原型图/截图），替代 Excel 提报 |
| FR-REQ-02 | 需求全生命周期状态机 | P0 | 状态：草稿→待分析→分析完成→评审中→已排期→开发中→测试中→待验收→已上线→已关闭/已打回；每次流转留痕（操作人、时间、意见） |
| FR-REQ-03 | 主/子需求树 | P0 | 支持一个业务需求拆分为多个系统需求（子需求），子需求可分属不同系统/团队，父需求进度自动汇总 |
| FR-REQ-04 | 业务需求与系统需求关联 | P0 | 双向关联与追溯：从业务需求可查看全部系统需求、代码变更、测试结果；从代码可反查需求来源 |
| FR-REQ-05 | 需求评论与@协作 | P1 | 需求详情页支持评论、@人、变更通知（IM/邮件） |
| FR-REQ-06 | 需求导入与 TAPD 下线迁移 | P1 | 支持存量 TAPD 需求 / Excel 一次性批量导入；迁移完成后 TAPD 正式下线（保留只读归档导出），不做双轨同步 |
| FR-REQ-07 | 需求变更管理 | P1 | 已排期需求发生变更时走变更流程，记录变更影响（工时/范围），通知相关人 |

**验收标准（示例，FR-REQ-01）**：业务方无需培训即可在 5 分钟内完成一条需求提交；必填项缺失有明确提示；提交后自动进入"待分析"并通知产品经理。

### 4.2 模块二：AI 可行性分析 Agent

| 编号 | 需求 | 优先级 | 说明 |
| --- | --- | --- | --- |
| FR-ANA-01 | 业务可行性分析 | P0 | Agent 基于需求描述 + 平台内历史需求/业务知识库，输出：业务合理性、与现有功能的重叠或冲突、相似历史需求 |
| FR-ANA-02 | 代码可行性分析 | P0 | Agent 检索关联代码库（RAG + 代码索引 + 调用链追踪），输出：涉及的系统/服务/模块清单、现有逻辑摘要、与新需求的冲突点、改动面评估 |
| FR-ANA-03 | 需求复杂度评估 | P0 | 输出复杂度分级（S/M/L/XL）及依据：涉及系统数、改动模块数、数据变更、外部依赖，并给出人天区间参考 |
| FR-ANA-04 | 系统范围识别 | P0 | 自动识别需求涉及的系统边界与上下游依赖，生成影响面图谱 |
| FR-ANA-05 | 分析报告生成与导出 | P0 | 汇总以上分析为结构化报告（结论、依据、风险、建议），支持在线查看与导出 PDF/Markdown，作为评审会材料 |
| FR-ANA-06 | 人工反馈闭环 | P1 | 产品/开发可对分析结论标注"准确/不准确+原因"，反馈用于优化提示词与知识库 |
| FR-ANA-07 | 增量重分析 | P1 | 需求补充信息后可触发重新分析，保留历史版本对比 |

**关键依赖**：代码知识库（见模块八）。分析质量取决于代码索引的覆盖度与新鲜度。

### 4.4 模块四：AI 需求与设计生成

| 编号 | 需求 | 优先级 | 说明 |
| --- | --- | --- | --- |
| FR-GEN-01 | 系统需求文档生成 | P0 | 基于业务需求 + 可行性分析结果，AI 生成系统需求文档（功能点、接口变更、数据模型变更、边界条件），产品经理在线修订后定稿 |
| FR-GEN-02 | 概要设计生成 | P1 | 基于系统需求 + 代码上下文生成概要设计（时序图、模块职责、接口契约草案），供开发评审修订 |

### 4.5 模块五：AI 编码与云开发

| 编号 | 需求 | 优先级 | 说明 |
| --- | --- | --- | --- |

| FR-DEV-02 | 云开发工作台 | P1 | 浏览器内 IDE（基于 code-server / Cloud Workspace），预置项目环境与 AI 助手，开发人员在线审查、修改、调试 AI 生成的代码 |

| FR-DEV-05 | 工作区安全隔离 | P0 | 每个 AI 编码任务运行在独立容器/工作树，资源限额，禁止越权访问其他项目代码 |

## 5. 非功能需求

| 编号 | 类别 | 需求 |
| --- | --- | --- |
| NFR-01 | 性能 | 常规页面响应 ≤ 1s；AI 可行性分析异步执行，P95 ≤ 30 分钟（含大型仓库检索）；分析任务支持进度推送（SSE/WebSocket） |
| NFR-02 | 可用性 | 核心服务可用性 ≥ 99.5%；AI 服务降级策略：模型不可用时流程可人工继续，不阻塞需求流转 |
| NFR-03 | 并发 | 支持 500 注册用户、100 并发在线、20 个并发 AI 分析/编码任务（可水平扩展） |
| NFR-04 | 安全 | 已决策使用外部 LLM API：代码/需求数据出网前必须经过脱敏网关（密钥、敏感字段、个人信息过滤）并通过安全合规评审，供应商协议须约定数据不用于训练、限期删除；平台自身数据不出私有网络边界；密钥统一托管；工作区容器隔离与资源限额 |
| NFR-05 | 数据 | 需求与审计数据保留 ≥ 3 年；每日备份，RPO ≤ 24h，RTO ≤ 4h |
| NFR-06 | 可观测 | 全链路日志/指标/追踪；AI 调用全量记录（供成本分析与问题回溯） |
| NFR-07 | 成本 | Token 成本按项目/Agent 维度可核算，支持配额与告警 |
| NFR-08 | 兼容 | 主流浏览器（Chrome/Edge/Safari 最新两个大版本）；云开发工作台建议 Chrome |

## 6. 技术架构（概要）

### 6.1 架构分层

```
┌──────────────────────────────────────────────────────┐
│  前端层  Nuxt/Vue Web 应用 + 云开发工作台(code-server)   │
├──────────────────────────────────────────────────────┤
│  应用服务层  需求服务 / 流程引擎 / 评审服务 / 报表服务      │
│             用户权限 / 通知 / 集成网关(GitLab·CI·IM)     │
├──────────────────────────────────────────────────────┤
│  AI 编排层  Agent Runtime(任务队列/工具调用/MCP)          │
│    ├ 可行性分析 Agent   ├ 需求/设计生成 Agent             │
│    ├ 编码 Agent         ├ 测试 Agent                    │
│    └ 翻译 Agent         └ 纪要/缺陷定位 Agent             │
├──────────────────────────────────────────────────────┤
│  知识层  代码索引/向量库 / 业务Wiki / 调用链与字段溯源       │
├──────────────────────────────────────────────────────┤
│  基础设施  PostgreSQL / Redis / 对象存储 / 向量数据库      │
│           K8s(Agent沙箱/工作台容器) / CI-CD / 监控告警     │
└──────────────────────────────────────────────────────┘
```

### 6.2 关键技术决策点

1. **LLM 供给方式 ✅ 已决策（2026-07-07）：外部 API**。全部 AI 能力经外部 LLM API 供给，不采购 GPU、不做私有化部署。硬性前置：M1.1 完成安全合规评审并落地脱敏网关；按任务分级用模型（分析/编码用强模型，翻译/摘要用性价比模型）。
2. **Agent 框架**：自研工具循环（可复用本仓库 trace_agent 模式）+ MCP 标准化工具接入。
3. **流程引擎**：需求流转内置于平台（完全替代 TAPD），采用轻量自研状态机——状态模型自主可控，才能让每个流转节点与 AI 产物（分析报告、文档、代码、测试结果）原生绑定；编排类长任务用任务队列，暂不引入 Temporal/Camunda。
4. **代码索引**：复用 deepwiki-open 已有的仓库解析、embedding（text-embedding-v4）、wiki 生成与 field_trace 能力作为知识层底座，可显著降低模块八的研发成本。
5. **云开发工作台 ✅ 已决策（2026-07-07）：自建**。基于 code-server/OpenVSCode + K8s 按需拉起，预置项目基础镜像与 AI 助手，不采购商业云 IDE。

### 7.1 阶段划分

### 7.2 关键路径与依赖

- **代码知识库（M1.3）是可行性分析、编码、测试三大 Agent 的共同前置**，必须一期落地并持续保鲜；
- LLM 已定为外部 API 供给，**安全合规评审与脱敏网关须在 M1.1 完成**，否则阻塞全部 AI 功能；
- 三期 AI 编码依赖 GitLab/CI 集成与容器沙箱平台，需提前与运维排期；
- TAPD 已决策下线：二期完成存量数据一次性迁移，需提前与各团队确认切换窗口与历史数据归档口径。

### 7.3 试点与推广策略

1. **试点先行**：一期以**银河开单系统**为试点业务域，自 M1.5 起该域全部新需求在平台提报与流转；
2. **直接切换**：不做与 TAPD 的长期双轨——试点域新需求直接走平台内置流转，二期完成存量数据一次性迁移后 TAPD 正式下线（保留只读归档导出）；
3. **逐步放权**：AI 产出始终"AI 起草 + 人工把关"，采纳率达标后再扩大 AI 自动化范围；
4. **运营机制**：每期设专职平台运营（可由产品兼任）收集反馈，双周迭代。
