## 1. 一期目标与验收标准（DoD）

**一句话目标**：银河开单系统的新需求全部在平台提报与流转，AI 可行性分析报告在真实评审会上被实际使用。

验收清单（全部满足才算一期完成）：

- [ ] 银河开单系统全部**新需求**在平台提报，走完整状态机流转（草稿→待分析→分析完成→评审中→已排期），全程留痕
- [ ] AI 可行性分析报告（业务可行性 / 代码可行性 / 复杂度 / 系统范围）产出耗时 ≤ 2 小时（一期 KPI）
- [ ] 至少 1 场真实评审会使用 AI 分析报告作为评审材料，评审结论在平台记录并驱动状态流转
- [ ] 主/子需求树与业务-系统需求双向关联可用
- [ ] RBAC 权限 + SSO 登录可用，试点相关人员全部开通
- [ ] 银河开单代码库接入完成：代码导航工具（MCP）可检索、模块级 repo map 生成，代码合入后工作树与 repo map 增量更新生效
- [ ] 脱敏网关上线：代码/需求数据出网前经过脱敏，安全合规评审结论落地
- [ ] 分析报告标注检索时间与代码出处引用（真实文件/行号，防误导评审的底线要求）

## 3. 快速落地策略

一期能压进 9 周，靠以下四条，缺一不可：

### 3.1 复用 deepwiki-open 存量能力（最大的加速器）

M1.3 代码知识库名义上排 5 周，实际是"接入 + 调优"而非从零开发。本仓库已有可直接复用的资产：

| 已有能力 | 位置 | 一期用法 |
| --- | --- | --- |
| 仓库拉取与解析管道（embedding 管道保留作对照） | `api/` 数据管道 | 接入银河开单代码库：工作树同步 + repo map 生成；embedding 管道用于 W3 检索策略 A/B |
| Wiki 生成流水线 + 后台任务系统（两阶段规划、增量更新、任务状态推进） | `api/wiki_generator.py`、`api/wiki_jobs.py` | 任务框架直接套用到"可行性分析任务"：异步执行、状态查询、超时处理 |
| 字段/调用链追踪 Agent（工具循环 + MCP） | `api/trace_agent.py`、`api/trace_tools.py`、`api/mcp_server.py` | 其工具集（grep/读文件/调用链）即 FR-KB-02 代码导航工具层，经 MCP 暴露给分析 Agent；工具循环模式直接复制到 FR-ANA-02/04 |
| Nuxt 4 前端骨架（页面、组件、composables） | `web/` | 平台前端在同一技术栈上扩展，不换栈 |

**决策**：一期平台直接在 deepwiki-open 代码库上演进，不另起新仓库——省去脚手架、CI、部署链路的重建成本。

**工程结构（已决策）**：单仓多服务，`web/` 为平台统一前端，后端按模块拆分服务：

| 目录 / 服务 | 定位 | 一期职责 | 端口 |
| --- | --- | --- | --- |
| `web/` | 统一前端 + Nitro BFF 网关 | 新增需求管理/评审/分析报告 UI；现有 wiki、trace 页面作为「知识库模块」并入平台导航（复用现有 Dashboard 布局与 BFF 代理模式）；BFF 统一鉴权与路由 | 3001 |
| `api/`（存量演进） | AI 与知识库服务（模块二/八） | 代码库接入与工作树同步、代码导航工具（MCP）、repo map、可行性分析 Agent 与分析任务（复用 wiki_jobs 模式）、调用链追踪、报告生成 | 8001 |
| `services/requirement/`（新建） | 需求域服务（模块一/三） | 需求 CRUD、状态机与留痕、主/子需求树、双向关联、评审协作、通知（MySQL） | 8002 |
| `services/identity/`（新建） | 身份与系统管理服务（模块十一） | SSO/LDAP、用户/角色 RBAC、项目空间、审计日志（含 AI 调用记录归集）（MySQL） | 8003 |

服务约定：前端只面向 BFF 一个入口；服务间调用走内部 JWT；每服务独立 MySQL schema 与迁移脚本，**禁止跨库 join**，跨域数据经 API 聚合。评审与需求同属一个流转域，故并入 requirement 服务，不单独拆。

### 3.2 AI-First 工法（人是审查者，不是打字员）

- **设计**：PM 用 AI 生成原型与 PRD 初稿 → 人工评审定稿；A1 用 AI 生成数据模型与 API 契约草案 → 架构评审定稿。
- **编码**：按模块写任务提示词（需求 + 契约 + 代码上下文）→ AI 生成代码 → 架构师 review 合入。**AI 生成的代码必须人工 review 才可合入**，无例外。
- **测试**：AI 依据契约生成接口用例与自动化脚本 → QA1 把关补边界/异常 → 入 CI。
- **资产化**：所有任务提示词、评测集、踩坑记录入库（repo 内 `docs/prompts/`），既降低单点风险，也是平台 FR-ADM-03 提示词管理的第一批真实素材。
- **节奏**：trunk-based，骨架先行，每日合入、CI 全绿是硬门禁；避免长分支。

### 3.3 评测集先行（Agent 质量不靠上线赌）

第 1 周就由 PM 收集 **10~20 条银河开单历史需求**（含当时的人工评估结论与实际落地情况）构建黄金评测集。此后 AI1 每一轮提示词/检索迭代都跑评测集打分（分析结论与历史事实的一致率），质量曲线可见、可汇报。**没有评测集，"分析准确率"就无从谈起，评审会也不敢用报告。**

### 3.4 双主线并行，第 5 周汇合

- **平台线**（A1+A2+PM）：需求管理中心 → 权限 → 评审协作，纯工程确定性高，先行封版。
- **AI 线**（AI1+QA1）：知识库接入 → 检索调优 → 分析 Agent → 报告生成，不确定性高，独立迭代不阻塞平台线。
- 两线在 W5 通过"分析任务 API"汇合集成（接口契约 W2 就定死，谁先做完谁先 mock）。

## 7. 关键技术落地要点

1. **状态机自研**（admin.md 6.2 决策 3）：状态与流转动作表驱动，每次流转写审计事件（操作人/时间/意见/关联 AI 产物 ID），一期就把"节点绑定 AI 产物"的数据结构立住——这是二期起流转上下文复用的地基。
2. **分析任务 = 长任务**：直接套 wiki_jobs 的 job 模型（排队/运行/成功/失败/超时 + 进度事件），前端 SSE 订阅；LLM 不可用时任务失败但流程可人工继续（NFR-02 降级要求）。
3. **脱敏网关先行于代码出网**：Agent 探索读取的代码片段出网前必须过网关（密钥/凭证/敏感配置正则 + 规则集可配置）——agentic 方式只有被探索片段出网、无需全量代码过 embedding API，安全面比全量索引更窄；安全评审未出结论前，仅用内部脱敏样本调试链路。
4. **报告可信度三件套**：结论必须携带①检索时间戳、②代码出处（真实文件/行号引用，agentic 直读天然成立）、③置信度标注——评审会敢用报告的前提。
5. **模型分级**：可行性分析/复杂度评估用强模型；检索重排、摘要类用轻模型控成本（Token 配额按 FR-ADM-03 最小集管理）。
6. **服务边界即模块边界**：requirement（模块一/三）、identity（模块十一）、api = AI 与知识库（模块二/八）；前端仅面向 Nitro BFF 单入口，鉴权在 BFF 统一做（会话换内部 JWT 下发给各服务）；分析任务由 requirement 服务发起调用 api 服务，结果回写走事件/回调，避免双向强耦合。
